# MCP Sentinel — Sub-Agent Orchestration Protocol

_Authority: this doc is loaded via CLAUDE.md reference at session start. Orchestrators and sub-agents both follow it._
_Established: wave 1 of Phase 1 migration (2026-04-21). Persisted after observing a 32-minute CI hang caused by 11 parallel agents each running the full analyzer suite._

## Why this exists

Without this protocol, parallel migration work blows up in three predictable ways:

1. **Test duplication** — 11 agents × full analyzer suite (1791 tests) = 11× waste.
2. **Shared-file conflicts** — when 5 agents each add imports to `rules/index.ts` and bump the regex baseline, every rebase produces merge conflicts.
3. **Runaway iteration** — agents that "fix until green" can spin on transient failures for 30+ minutes before surfacing a blocker the orchestrator could resolve in 2 minutes.

Wave 1 adopted this protocol operationally and cut per-wave wall time from ~50min to ~20min.

## Ground rules

### 1. One rule per worktree

Every sub-agent gets a dedicated git worktree (`isolation: "worktree"`) and works on exactly one rule. Never ask a single agent to migrate two rules — even "related" ones. Keeps PRs atomic, rebases cheap, and sub-agent reasoning focused.

### 2. Agent test commands (allowed list — no others)

Sub-agents run **only** these in their worktree:

```bash
# Typecheck just the analyzer package (not the full monorepo)
pnpm --filter=@mcp-sentinel/analyzer typecheck

# Run ONLY the new rule's own test file
pnpm --filter=@mcp-sentinel/analyzer vitest run \
  src/rules/implementations/<rule-dir>/__tests__/index.test.ts

# Strict guards (the two that catch architectural drift)
ANALYZER_STATIC_GUARD_STRICT=true pnpm --filter=@mcp-sentinel/analyzer vitest run \
  __tests__/no-static-patterns.test.ts \
  __tests__/charter-traceability.test.ts
```

**Explicitly forbidden for agents**:
- `pnpm test` at repo root
- `pnpm turbo test`
- Full analyzer suite (`pnpm --filter=@mcp-sentinel/analyzer test`)
- Any cross-package integration test

Agents trust that the orchestrator will run the full suite on the merged tree.

### 3. Files agents must NOT touch

Agents producing a new rule directory MUST NOT modify:

| Path | Owned by | Reason |
|---|---|---|
| `docs/census/*` | orchestrator | Regenerated by `pnpm rule:census` — per-agent edits produce merge conflicts on every rebase |
| `docs/plans/*` | orchestrator | Status updates happen after merge |
| `packages/analyzer/src/rules/index.ts` | orchestrator | Single bottleneck file; serialised adds prevent rebase cascades |
| `docs/census/regex-baseline.json` | orchestrator | Baseline ratchet is deferred until all wave PRs land |
| Legacy detector files shared across migrations (e.g. `k-compliance-v2.ts`) | orchestrator | Agents propose a removal block; orchestrator applies it |

If an agent needs a baseline bump to pass the strict guard, the orchestrator raises the baseline in the cleanup commit — never in the rule PR.

### 4. Agents do NOT iterate on failure

If tests fail in the agent's worktree, the agent reports the failure with the exact error and stops. It does NOT:
- re-run the same command hoping for a different result
- start editing unrelated files to work around a failure
- keep trying variants until something passes

The orchestrator triages post-merge. A failing PR that surfaces quickly beats a silent agent that spins for 30 minutes.

### 5. Orchestrator cleanup commit (one per wave)

After all wave PRs merge, the orchestrator runs exactly one cleanup commit on main (or a cleanup PR if main is protected):

```
1. Add all new `import "./implementations/<rule-dir>/index.js";` lines to rules/index.ts
2. pnpm rule:census   (regenerates docs/census/)
3. Update docs/plans/full-migration-plan-2026-04-20.md with chunk status
4. Delete any now-empty legacy detector files the wave obsoleted
5. Bump docs/census/regex-baseline.json if the wave's structural migrations dropped the count
6. Fix any compilation errors that only surface when all the new rules coexist (e.g. duplicate rule IDs)
```

Gate the cleanup commit on: `pnpm typecheck` + `pnpm turbo test` + `ANALYZER_STATIC_GUARD_STRICT=true pnpm vitest run __tests__/no-static-patterns.test.ts __tests__/charter-traceability.test.ts`.

## Briefing template for sub-agents

Every agent prompt MUST include:

```
Working directory: <absolute path to worktree>
Base commit:       <exact SHA, e.g. 949cc86>
Target rule:       <rule id + name, e.g. K14 Agent Credential Propagation>
Reference:         packages/analyzer/src/rules/implementations/k1-absent-structured-logging/
Standard:          docs/standards/rule-standard-v2.md
CVE manifest:      docs/cve-manifest.json (add entry if rule cites a CVE not already listed)

Deliverables (all required):
- CHARTER.md       ≤120 lines, ≥3 lethal edge cases, interface_version: "2.0",
                   evidence_contract declared
- gather.ts        Deterministic fact-collection only. No regex literals, no
                   string arrays > 5. Structured Location on every fact.
- verification.ts  Named VerificationStep factories, each returns a step whose
                   `target` field is a Location (not a prose string).
- index.ts         class Xxx implements TypedRuleV2; EvidenceChainBuilder with
                   source + propagation* + sink + mitigation + impact links;
                   confidence cap from CHARTER; registerTypedRuleV2 at module level.
- data/*.ts        Vocabulary / target-lookup tables as typed records.
- __fixtures__/    ≥3 true-positive + ≥2 true-negative TypeScript files.
- __tests__/index.test.ts  Unit tests covering each CHARTER lethal edge case + chain shape.

Test commands (run these AND ONLY these before claiming done):
- pnpm --filter=@mcp-sentinel/analyzer typecheck
- pnpm --filter=@mcp-sentinel/analyzer vitest run src/rules/implementations/<rule-dir>/__tests__/index.test.ts
- ANALYZER_STATIC_GUARD_STRICT=true pnpm --filter=@mcp-sentinel/analyzer vitest run __tests__/no-static-patterns.test.ts __tests__/charter-traceability.test.ts

DO NOT modify:
- docs/census/*
- docs/plans/*
- packages/analyzer/src/rules/index.ts
- docs/census/regex-baseline.json
- Any legacy detector file (<name>-v2.ts) beyond removing the rule's own class

On failure: report the exact error and STOP. Do not retry, do not edit around.

Commit message: feat(phase-1): migrate <RULE_ID> (<name>) to Rule Standard v2
Branch name:    claude/phase-1/<rule-id-kebab>
Open PR:        yes, one PR per rule.
```

## Decision rules for the orchestrator

- **Wave size**: up to 11 parallel agents empirically works. Beyond that, rebase cascades become the bottleneck even with this protocol.
- **Cross-rule dependencies**: if rule A's shared infrastructure (e.g. `_shared/taint-rule-kit/`) is needed by rule B, do A in wave N, B in wave N+1. Never fan them out together.
- **Rebase cascade**: when wave PRs touch a shared file (plan doc, rules/index.ts), merge them one-at-a-time. Each merge triggers a re-rebase for the others. Budget ~5min per re-rebase.
- **Guard mode**: keep `no-static-patterns` and `charter-traceability` warn-only in CI until Phase 1 chunk 1.28, but sub-agents MUST run them in strict mode locally so the regression surfaces before PR.
- **Post-merge fixes**: if a rule introduces a bug only visible in the merged tree (e.g. K18's infinite-loop in wave 1), push the fix as a direct commit to main, not as a revert. Reverts confuse the plan-doc status.

## What to log after each wave

Append to `docs/plans/full-migration-plan-2026-04-20.md` "Completed in wave N" section:

| Chunk | Deliverable | Commit(s) |
|---|---|---|

Plus:
- net regex-literal delta
- total new tests
- any post-merge fix commits with one-line explanations

This is what makes the methodology reproducible across sessions.
